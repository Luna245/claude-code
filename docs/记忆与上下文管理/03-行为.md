# 03 - 行为篇：加载时机与全景架构

> [第 02 章](./02-结构.md)讲了"有什么"，这一章讲"什么时候出现在上下文里"。这是 Claude Code 的灵魂部分。

---

## 一、三类加载方式

所有组件按"何时进入上下文"分为三类：

```
              ┌─────────── 上下文窗口 ───────────┐
              │                                  │
始终在 ──→    │  ● System Prompt                 │
(每轮都带)    │  ● AGENTS.md / CLAUDE.md         │
              │  ● MCP 工具定义                  │
              │  ● Skill 元数据 (name + desc)    │
              │  ● Subagent 元数据               │
              │                                  │
按需进 ──→    │  ◐ Skill 正文（任务匹配时）       │
(触发才占)    │  ◐ Skill 引用的子文件             │
              │  ◐ MCP 工具调用结果              │
              │  ◐ 文件 Read 结果                │
              │                                  │
              └──────────────────────────────────┘

永不进 ──→    ✗ Skill 里的 Python/Shell 脚本源码
(只看输出)    ✗ Subagent 内部的工作上下文（隔离）
              ✗ MCP server 内部实现
              ✗ Hook 脚本本身
```

---

## 二、关键澄清：「加载一次」是个伪命题

很多人以为 CLAUDE.md "只在会话开始加载一次"——**这是错的**。

正确的理解需要分**两个层次**：

### 客户端层面：确实只读一次

```
Claude Code 启动时：
  ├── 读 ~/.claude/CLAUDE.md            ← 一次
  ├── 读 ./AGENTS.md                    ← 一次
  ├── 扫描 ./skills/ 目录                ← 一次
  ├── 读 mcp.json，启动 MCP server       ← 一次
  └── 把这些内容缓存在客户端内存里
```

到这里都没问题。**但是**——

### 模型层面：每条消息都重新发送

```
你发第 1 条消息：
  客户端从内存取出 CLAUDE.md，拼进请求，发送
  
你发第 2 条消息：
  客户端从内存取出 CLAUDE.md，再次拼进请求，再次发送
  
你发第 N 条消息：
  客户端从内存取出 CLAUDE.md，再次拼进请求，再次发送
```

**因为 LLM 是无状态的**（见 [01 章](./01-基础概念.md)），每次请求模型必须看到完整上下文。

## 三、Progressive Disclosure（渐进式披露）

这是 Skill 设计最精巧的部分。**只有名字和描述始终在**，其余按需加载。

```
Level 1 — Front Matter（始终加载）
┌──────────────────────────────────────┐
│ ---                                  │
│ name: deploy-frontend                │
│ description: 部署前端到生产环境       │
│ ---                                  │
└──────────────────────────────────────┘
        ↓ Claude 判断当前任务匹配
Level 2 — SKILL.md 正文（按需）
┌──────────────────────────────────────┐
│ # 部署流程                            │
│ 1. 预检查...                         │
│ 2. 构建...                           │
└──────────────────────────────────────┘
        ↓ 正文引用了子文件
Level 3 — 子文件（选择性）
┌──────────────────────────────────────┐
│ checklist.md, rollback.md ...        │
└──────────────────────────────────────┘
        ↓ 需要执行操作
Level 4 — 脚本（执行，不加载）
┌──────────────────────────────────────┐
│ ./scripts/deploy.sh                  │
│ → 通过 Bash 工具调用                  │
│ → 只有输出进入上下文，源码不进         │
└──────────────────────────────────────┘
```

**为什么这么设计**：50 个 Skill 的库不会拖慢 Claude，因为只占 50 × 100 = 5000 token 的元数据。但**一个 5000 行的 CLAUDE.md 会**——它整文件始终在上下文里。

## 四、完整请求时序

一次"用户发消息→收到回复"的完整流程：

```
[用户输入消息]
       │
       ▼
┌──────────────────────────────────────────────┐
│ Claude Code 客户端                            │
│                                              │
│ 1. 从内存取出已读的 CLAUDE.md / skill 表      │
│ 2. 拼装上下文：                                │
│    [System Prompt]                           │
│    [CLAUDE.md / AGENTS.md]                   │
│    [MCP 工具定义]                             │
│    [Skill 元数据列表]                         │
│    [Subagent 元数据列表]                      │
│    [历史对话 N 轮]                            │
│    [本轮用户消息]                             │
│ 3. 给前缀打 cache_control 标记                │
│ 4. 发送 HTTPS 请求（完整上下文）              │
└────────────────────┬─────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────┐
│ Anthropic 服务端                              │
│                                              │
│ 5. 查 Prompt Cache：                          │
│    ├─ 命中 → 复用之前的 KV 状态                │
│    └─ 未命中 → 重新 prefill 整个前缀           │
│ 6. 调度到推理集群                              │
└────────────────────┬─────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────┐
│ Claude 模型推理                               │
│                                              │
│ 7. 看到完整上下文，决定怎么回复                 │
│ 8. 可能要求调用：                              │
│    ├─ 读取 Skill 正文（"我需要 deploy 技能"）  │
│    ├─ 调用 MCP 工具                            │
│    ├─ 启动 Subagent                            │
│    └─ 执行 Bash 命令                           │
└────────────────────┬─────────────────────────┘
                     │ 流式返回
                     ▼
┌──────────────────────────────────────────────┐
│ Claude Code 客户端                            │
│                                              │
│ 9. 收到回复，执行工具调用                      │
│ 10. 工具结果作为新消息再走一遍 5~8             │
│ 11. 最终结果展示给用户                        │
│ 12. 把这一轮的所有内容存进会话历史              │
└──────────────────────────────────────────────┘
```

## 五、全景架构图：四层视角

```
┌─────────────────────────────────────────────────────────────────┐
│                  ① 磁盘层 (Disk) ─ 客户端启动时只读一次          │
│                                                                 │
│  ~/.claude/CLAUDE.md      ./AGENTS.md      ./skills/*/SKILL.md  │
│  ~/.claude/agents/*.md    ./mcp.json       ./scripts/*.sh       │
└───────────────────────────────┬─────────────────────────────────┘
                                │ 启动时读取
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│             ② Claude Code 客户端 (Client) ─ 内存中               │
│                                                                 │
│  保存所有读到的文件内容                                          │
│  维护会话历史                                                    │
│  每次发消息时：                                                  │
│    把"始终在的" + 历史 + 新消息拼装                              │
│    标记 cache_control 缓存断点                                  │
└───────────────────────────────┬─────────────────────────────────┘
                                │ HTTPS API（每轮完整发送）
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│             ③ Anthropic 服务端 ─ Prompt Cache 层                 │
│                                                                 │
│  按 cache_control 哈希前缀                                       │
│  命中 → 跳过 prefill 计算，按 10% 价格计费                       │
│  未命中 → 全量 prefill，全价计费                                 │
│  TTL: 5 分钟（默认）/ 1 小时（订阅或显式指定）                    │
└───────────────────────────────┬─────────────────────────────────┘
                                │ 解码后的 KV 传给模型
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                  ④ 模型 (Claude) ─ 无状态                        │
│                                                                 │
│  每轮看到完整 token 序列                                         │
│  不"记得"任何东西，所有上下文都靠客户端重传                       │
│  推理生成下一个 token                                            │
└─────────────────────────────────────────────────────────────────┘
```

## 六、上下文窗口的内容布局

一次完整 API 调用，上下文里的内容是这样排列的：

```
┌─────────────────────────────────────────────────────┐  ▲
│ ① System Prompt（Anthropic 内置 + Claude Code）      │  │ 可缓存前缀
├─────────────────────────────────────────────────────┤  │ (cache_control)
│ ② AGENTS.md / CLAUDE.md（项目约定）                  │  │
├─────────────────────────────────────────────────────┤  │ 只要内容不变
│ ③ MCP 工具定义（每个 server 几千~上万 token）         │  │ 后续请求
├─────────────────────────────────────────────────────┤  │ 命中缓存
│ ④ Skill 元数据（name + description，约 100/个）       │  │
├─────────────────────────────────────────────────────┤  │
│ ⑤ Subagent 元数据                                    │  ▼
├═════════════════════════════════════════════════════┤
│ ⑥ 历史对话（动态累积）                                │  ▲
│    ├─ 用户消息 1                                     │  │ 动态部分
│    ├─ 助手回复 1                                     │  │ 不缓存
│    ├─ 触发的 Skill 正文（如有）                       │  │ 每轮新增
│    ├─ MCP 工具调用 & 结果                            │  │
│    ├─ ...                                            │  │
│    └─ 用户消息 N（本轮新消息）                        │  ▼
└─────────────────────────────────────────────────────┘
```

**关键**：上半部分（①~⑤）是**稳定前缀**，可被 Prompt Cache 复用。下半部分（⑥）每轮都在变，缓存不到。

## 七、各组件成本/性能影响速查

| 组件 | 启动 token 占用 | 每轮重传 | 可缓存 |
|------|--------------|--------|------|
| System Prompt | 几千 | 是 | 是 ✅ |
| AGENTS.md / CLAUDE.md | 文件全长 | 是 | 是 ✅ |
| MCP 工具定义 | 几千~几万/server | 是 | 是 ✅ |
| Skill 元数据 | ~100/个 | 是 | 是 ✅ |
| Skill 正文 | 0（按需才进） | 触发后是 | 部分 ⚠ |
| MCP 调用结果 | 0（调用才有） | 之后历史里 | 不缓存 ❌ |
| 历史对话 | 0 起步 | 累积重传 | 不缓存 ❌ |

## 八、四种"加载"的真实含义对照

| 你以为的 | 实际发生的 | 在哪里发生 |
|---------|----------|-----------|
| "CLAUDE.md 会话开始加载一次" | 磁盘读一次，但每轮都重发给模型 | 客户端读文件 ✅ / 模型处理 ❌ |
| "Skill 按需加载" | 元数据始终在，正文匹配时才被读 + 拼入上下文 | 真·按需 ✅ |
| "MCP 启动时加载" | 所有工具定义启动时拼入上下文，从此始终在 | 模型层 ✅ |
| "Prompt cache 缓存了上下文" | 缓存了**计算结果**（KV），不是省 token 数量 | 服务端计算层 ✅ |

## 九、为什么"上下文越大越慢/越贵"

```
上下文 1k token：
  prefill 时间：~50ms
  成本：很低
  注意力：聚焦

上下文 50k token：
  prefill 时间：~2s
  成本：50 倍
  注意力：开始稀释

上下文 150k token（接近上限）：
  prefill 时间：~6s
  成本：150 倍
  注意力：明显降智
  风险：很容易触顶
```

所以**"上下文管理"的核心是**：
1. 该常驻的就常驻（CLAUDE.md），别长得离谱
2. 偶尔用的做成 Skill，按需加载
3. 大量数据用 MCP/Subagent，别全塞进主上下文
4. 用 Prompt Cache 把常驻部分的**计算成本**降下来
