# 01 - 基础概念

> 在讨论 Claude Code 怎么"记忆"之前，先搞清楚 LLM 本身是怎么工作的。这一章只讲底层原理，不涉及任何具体配置。

---

## 一、LLM 是无状态的

**核心事实**：Claude（以及任何大语言模型）**没有内置记忆**。

每次你发一条消息，本质上是这样的：

```
你的客户端                               Anthropic 服务端
    │                                        │
    │  把"完整对话历史 + 这次的新消息"        │
    │  打包成一坨 token 序列                  │
    ├───────────────────────────────────────→│
    │                                        │
    │                                  模型处理这坨 token
    │                                        │
    │←───────────────────────────────────────│
    │           回复内容                       │
    │                                        │
```

**关键点**：模型每次看到的就是一坨完整的 token 序列，它不"记得"上次说过什么。所谓"上下文"，是客户端帮你**每次都重新拼好发过去**的。

这就引出一个重要推论：**任何"会话级"的概念，都是客户端层面的抽象**。从模型视角看，每次请求都是一锤子买卖。

---

## 二、上下文窗口（Context Window）

每个模型有一个**最大输入容量**，叫上下文窗口，单位是 **token**。

```
┌────────────────────────────────────────────┐
│  上下文窗口 (e.g. 200,000 tokens)           │
│                                            │
│  ┌──────────────────────────────────────┐  │
│  │ 这次请求要塞进去的所有内容            │  │
│  │ - 系统提示                            │  │
│  │ - 项目约定                            │  │
│  │ - 工具定义                            │  │
│  │ - 历史对话                            │  │
│  │ - 当前消息                            │  │
│  └──────────────────────────────────────┘  │
│                                            │
│  超出上限 → 报错或被截断                    │
└────────────────────────────────────────────┘
```

**为什么这个数字重要**：
- 决定了你能塞多少历史、多少文档、多少工具定义
- 接近上限时会**降智**（模型注意力被稀释，效果下降）
- 直接影响**每次请求的成本**

---

## 三、Token 是什么

Token 不是字符也不是单词，是 **BPE 分词器**切出来的语言单位。

| 文本 | Token 数 |
|------|---------|
| `hello` | 1 |
| ` hello`（前面带空格） | 1（空格和词合并） |
| `hello world` | 2 |
| `        `（连续 8 空格） | 1~2 |
| `你好` | 2~3 |
| 一段 1000 字英文 | 约 250 token |
| 一段 1000 字中文 | 约 600~800 token |

**经验法则**：
- 英文：**1 token ≈ 4 字符**
- 中文：**1 字 ≈ 1~2 token**
- 代码：**1 token ≈ 3~4 字符**（更细碎）

---

## 四、客户端、服务端、模型——三个不同的"角色"

讨论 Claude Code 时，要时刻分清这三层：

```
┌─────────────────────────────────────────────────────────┐
│  客户端 (Client) — Claude Code 自己                      │
│                                                          │
│  • 读你磁盘上的 CLAUDE.md、Skill 文件                    │
│  • 维护"会话"概念（其实只是它在记历史）                   │
│  • 每次帮你把上下文拼好发出去                            │
│  • 处理工具调用结果、显示给你看                          │
└──────────────────────────┬──────────────────────────────┘
                           │ HTTPS API
                           ▼
┌─────────────────────────────────────────────────────────┐
│  服务端 (Anthropic Server) — API 入口                    │
│                                                          │
│  • 接收请求                                              │
│  • Prompt Cache 在这一层（看下文）                       │
│  • 调度到推理集群                                        │
│  • 返回响应                                              │
└──────────────────────────┬──────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  模型 (Model) — Claude 本身                              │
│                                                          │
│  • 看到一坨完整 token                                    │
│  • 算出下一个 token                                      │
│  • 没有记忆、没有状态                                    │
└─────────────────────────────────────────────────────────┘
```

很多迷惑都来自混淆这三层。比如"CLAUDE.md 加载一次"——这在**客户端层面**是真的（文件 I/O 只读一次），但在**模型层面**是假的（每次还是要重传）。

---

## 五、为什么 Claude Code 看起来"有记忆"

它根本没有，全靠**客户端帮你管历史**：

```
你说："帮我看看 user.py"
   ↓
客户端发送：[系统提示] [CLAUDE.md] [用户: 帮我看看 user.py]
   ↓
模型回复：[Claude: 这个文件有个 bug...]
   ↓
客户端把这条回复存进内存
   ↓
你说："修一下"
   ↓
客户端发送：[系统提示] [CLAUDE.md] [用户: 帮我看看 user.py] 
           [Claude: 这个文件有个 bug...] [用户: 修一下]
   ↓
（注意：第一轮的所有内容又被原封不动重发了一遍）
```

每一轮，客户端都把**之前所有内容 + 新消息**重新打包发给模型。模型每次都是从零看一遍。

---

## 六、为什么这样设计

无状态有它的好处：

- **横向扩展容易**：任何一台机器都能处理任何一个请求
- **容错强**：单台机器挂了不影响"会话"
- **责任清晰**：客户端管历史，服务端管计算

代价就是：**重复传输 + 重复计算**。Prompt Cache 就是用来缓解后者的（[第 04 章](./04-prompt-caching.md)详细讲）。

---

## 关键术语速查

| 术语 | 含义 |
|------|------|
| Context Window | 一次请求能塞下的最大 token 数 |
| Token | 语言切分的最小单位，不是字符也不是单词 |
| Stateless | 模型无状态，每次请求独立处理 |
| Prefill | 服务端处理输入 token 的阶段 |
| Decode | 服务端生成输出 token 的阶段 |
| Prompt Cache | 缓存"前缀计算结果"以加速重复请求 |
