# 02 - 结构篇：Claude Code 的各组件

> 这一章只讲**"有哪些东西"**，不讲它们什么时候加载。加载时机看 [第 03 章](./03-行为.md)。

---

## 概览：六大扩展机制

Claude Code 通过这六种机制让 Claude 适配你的项目：

```
┌──────────────────────────────────────────────────────┐
│              Claude Code 扩展体系                     │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ① System Prompt    - Anthropic 内置 + Claude Code   │
│  ② CLAUDE.md/AGENTS.md - 项目级永久约定              │
│  ③ Skill            - 按需触发的工作流              │
│  ④ MCP Server       - 外部服务/工具接入             │
│  ⑤ Subagent         - 独立上下文的子任务            │
│  ⑥ Hook             - 生命周期事件触发的脚本        │
│                                                      │
└──────────────────────────────────────────────────────┘
```

各自解决不同问题：

| 组件 | 解决什么 | 类比 |
|------|---------|------|
| System Prompt | Claude 的基础人格、能力定义 | "出厂设置" |
| CLAUDE.md | 项目永久规则（"用 4 空格缩进"） | "项目 README" |
| Skill | 偶尔用到的工作流（"怎么发布"） | "操作手册" |
| MCP | 访问外部数据（数据库、GitHub） | "外接 API" |
| Subagent | 独立跑的专家子任务 | "外包给同事" |
| Hook | 自动触发的脚本 | "Git hook" |

---

## 一、System Prompt（系统提示）

由 **Anthropic 和 Claude Code** 一起决定，**你改不了**。

包含：
- Claude 的身份定义
- 默认工作模式
- 安全约束
- Claude Code 的工具使用规范

**特点**：
- 完全不可见、不可改
- 每次请求都会发送
- 通常几千 token

---

## 二、CLAUDE.md / AGENTS.md

**位置**：项目根目录或 `~/.claude/CLAUDE.md`（全局）

**作用**：项目级的**永久指令**，每次都会注入。

```markdown
# CLAUDE.md 示例

## 项目约定
- 使用 TypeScript strict 模式
- 测试用 vitest，不要用 jest
- 提交信息遵循 Conventional Commits

## 代码风格
- 函数式优先，避免 class
- 错误处理用 Result 类型，不要抛异常
```

**特点**：
- 始终加载，整文件进上下文
- 适合**变化不大的项目约定**
- **越长越费 token，越长越容易让 Claude 失焦**

**AGENTS.md vs CLAUDE.md**：
- 二者作用基本相同
- `AGENTS.md` 是 [agents.md 标准](https://agents.md/)，跨工具通用（Cursor、Windsurf 等也读）
- `CLAUDE.md` 是 Claude 专属
- 现代项目推荐用 `AGENTS.md`，必要时让 `CLAUDE.md` 仅做引用

---

## 三、Skill（技能）

**位置**：
- 项目级：`./skills/<skill-name>/SKILL.md`
- 用户级：`~/.claude/skills/<skill-name>/SKILL.md`

**作用**：**按需触发**的工作流知识，不用时不占上下文。

### Skill 文件结构

```
skills/
└── deploy-frontend/
    ├── SKILL.md          ← 主文件，YAML front matter + 说明
    ├── checklist.md      ← 引用的子文件
    ├── rollback.md       ← 引用的子文件
    └── scripts/
        └── deploy.sh     ← 可执行脚本（永远不进上下文）
```

### SKILL.md 写法

```markdown
---
name: deploy-frontend
description: 部署前端到生产环境的标准流程，包括预检查、构建、灰度、回滚
---

# 前端部署流程

## 预检查
1. 确认所有测试通过
2. 检查 staging 环境状态
...
```

**前置元信息（front matter）始终加载**，正文按需加载。这就是 [Progressive Disclosure 渐进式披露](./03-行为.md#progressive-disclosure)。

### Skill 的层级加载

| 层级 | 内容 | 何时加载 |
|------|------|---------|
| L1 | name + description（约 100 token） | 始终 |
| L2 | SKILL.md 正文 | 任务匹配时 |
| L3 | SKILL.md 里引用的子文件 | 用到时才读 |
| L4 | 脚本源码 | **永不加载，只执行看输出** |

---

## 四、MCP Server（Model Context Protocol）

**作用**：让 Claude 连接外部系统——数据库、GitHub、Jira、文件系统等。

### 配置位置

`~/.claude/mcp.json` 或项目级 `.mcp.json`：

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "ghp_..."
      }
    },
    "postgres": {
      "command": "uvx",
      "args": ["mcp-server-postgres", "postgresql://localhost/mydb"]
    }
  }
}
```

### MCP 的两类内容

```
MCP Server 提供：
├── Tools（工具）     - Claude 可以调用的函数
├── Resources（资源） - Claude 可以读取的数据
└── Prompts（提示词） - 预定义的提示模板
```

### MCP 的成本

**MCP 在上下文里很贵**：每个 server 的所有工具定义（名字、描述、参数 schema）都在会话开始时塞进上下文，从此**始终占用**。

| 配置 | 启动占用 |
|------|---------|
| 3 个 MCP server | 约 4 万 token |
| 5 个 MCP server | 约 5.5 万 token |
| 7 个 MCP server | 约 6.7 万 token（33% 上下文窗口） |

新版 Claude Code 2.1+ 支持 `context: fork`，让 MCP 只在子任务里加载，减轻主会话压力。

## 五、Subagent（子代理）

**位置**：`.claude/agents/<agent-name>.md` 或 `~/.claude/agents/<agent-name>.md`

**作用**：把任务委托给一个**有独立上下文窗口**的子 Claude，结果以摘要形式返回主会话。

### 定义示例

```markdown
---
name: code-reviewer
description: 资深代码审查员，关注安全和性能问题
tools: [Read, Grep, Bash]
mcp: [github]
model: claude-opus-4-8
---

你是一个严格的代码审查员。审查时关注：
1. 安全漏洞
2. 性能问题
3. 可维护性
...
```

### Subagent vs Skill 的关键区别

| 维度 | Skill | Subagent |
|------|-------|---------|
| 上下文 | 共享主会话 | **独立**上下文窗口 |
| 隔离性 | 弱 | 强 |
| 返回内容 | 全部对话进主上下文 | 只返回**摘要** |
| 适合场景 | 程序性知识 | 大块独立任务 |

**典型用法**：让 subagent 跑一遍代码审查或长文档分析，主会话只看到结论，不被中间过程污染。

## 六、Hook（钩子）

**位置**：`~/.claude/settings.json` 的 `hooks` 字段

**作用**：在 Claude Code 生命周期事件上自动跑脚本。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint --silent"
          }
        ]
      }
    ]
  }
}
```

**常见事件**：
- `PreToolUse` - 工具调用前
- `PostToolUse` - 工具调用后
- `Notification` - 收到通知时
- `Stop` - 模型停止生成时
- `SessionStart` - 会话启动

**特点**：
- Hook 本身**不进上下文**
- 但 hook 的输出（如果配置成 inject）会作为消息插入对话

---

## 各组件对照表

| 组件 | 文件位置 | 启动占 token | 加载方式 | 适合 |
|------|---------|------------|---------|------|
| System Prompt | 内置 | 几千 | 始终 | 不可改 |
| CLAUDE.md / AGENTS.md | 项目根 / 全局 | 文件全长 | 始终 | 永久约定 |
| Skill 元数据 | `skills/*/SKILL.md` 头部 | ~100/个 | 始终 | 让 Claude 知道有这个能力 |
| Skill 正文 | `skills/*/SKILL.md` | 0 | 按需 | 偶尔用到的工作流 |
| MCP 工具定义 | `mcp.json` | 几千~几万/server | 始终 | 外部系统接入 |
| MCP 调用结果 | - | 0 | 调用时 | 实际数据 |
| Subagent 元数据 | `agents/*.md` 头部 | 少量 | 始终 | 让 Claude 知道有这个专家 |
| Subagent 工作内容 | - | 0（隔离） | 调用时（独立上下文） | 大块委托任务 |
| Hook | `settings.json` | 0 | 触发时 | 自动化动作 |
